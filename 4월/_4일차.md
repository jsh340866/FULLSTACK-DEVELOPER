## ERD 정리
### 1. ERD 기본 개념
- **엔티티(Entity)**
  - 시스템에서 **저장·관리해야 하는 대상**을 의미.
  - ERD에서는 **직사각형**으로 표현.
  - 예시: `고객`, `사람`, `지문`, `장바구니`, `제품`, `직원`, `매장`, `부서`, `학생`, `강의`, `남성`, `여성` 등
- **관계(Relationship)**
  - 두 엔티티가 **어떤 의미로 연결되는지** 나타냄.
  - ERD에서는 **마름모(◇)** 로 표현.
  - 예시: `결혼`, `혼인`, `등록/소유`, `임시보관`, `주문`, `소속`, `연결`, `수강신청` 등
- **속성(Attribute)**
  - 엔티티가 가진 **세부 정보(컬럼)**.
  - Chen 표기에서는 **원(○)** 으로, 테이블 ERD에서는 **행(row)** 으로 표현.
  - 예시:
    - 사람: `이름`, `주소`, `주민등록번호(식별자)`
    - 지문: `지문ID`, `지문 이미지`
    - 남성/여성: `manId`, `womanId`, `이름`, `나이`, `주소` 등
- **키(Key)**
  - **PK (Primary Key)**: 행을 유일하게 식별하는 컬럼.
  - **FK (Foreign Key)**: 다른 테이블의 PK를 참조하는 컬럼.
  - 예시:
    - `남성` 테이블: `manId` → **PK**
    - `여성` 테이블: `womanId` → **PK**
    - `성혼` 테이블: `manId`, `womanId` → **PK, FK** (양쪽 테이블을 참조하는 교차 테이블)
### 2. 관계 차수(Cardinality)
- **1:1 (일대일)**
  - 한 엔티티의 한 행이, 다른 엔티티의 **오직 한 행**과만 연결.
  - 예시: 이론적으로 “남자 1명 ↔ 여자 1명” 형태의 `혼인` 관계.
- **1:N (일대다)**
  - 한 엔티티의 한 행이, 다른 엔티티의 **여러 행**과 연결.
  - 예시:
    - `사람` –(등록/소유)– `지문`  
      → 사람 1명은 여러 개의 지문을 가질 수 있음.
    - `고객` –(임시보관)– `장바구니`  
      → 고객 1명은 장바구니를 여러 개(또는 0개) 가질 수 있음.
    - `직원` –(소속)– `매장`  
      → 하나의 매장에 여러 직원이 소속될 수 있음 (또는 반대로 모델링).
- **N:M (다대다)**
  - 한 엔티티의 여러 행이, 다른 엔티티의 여러 행과 연결.
  - 예시:
    - `고객` –(주문)– `제품`  
      → 고객은 여러 제품을 주문하고, 제품도 여러 고객에게 주문될 수 있음.
    - `강의` –(수강신청)– `학생`  
      → 한 강의에 여러 학생이 수강, 한 학생도 여러 강의를 수강.
- **테이블 구현 시 N:M → 중간 테이블로 변환**
  - 예: `남성` –(성혼)– `여성`
    - `남성(manId PK, ...)`
    - `여성(womanId PK, ...)`
    - `성혼(manId PK,FK, womanId PK,FK, 결혼일자, ...)`
  - 예: `강의` –(수강신청)– `학생`
    - `학생(학생ID PK, ...)`
    - `강의(강의ID PK, ...)`
    - `수강신청(학생ID FK, 강의ID FK, 수강일시, ...)`
### 3. Chen ERD → 테이블(관계형 모델) 변환 패턴
- **엔티티(Entity) → 테이블(Table)**
  - Chen ERD에서 직사각형 엔티티 하나가 DB의 **테이블 하나**에 대응.
  - 예:
    - 엔티티 `고객` → 테이블 `Customer`
    - 엔티티 `제품` → 테이블 `Product`
- **속성(Attribute) → 컬럼(Column)**
  - 엔티티 주변의 원(속성)이 테이블의 **컬럼**이 됨.
  - 예:
    - 엔티티 `사람`의 속성 `이름`, `주소`, `주민등록번호`  
      → `Person(name, address, ssn, ...)`
- **1:N 관계 구현**
  - N 쪽 테이블에 **FK 컬럼**을 추가하는 방식이 일반적.
  - 예:
    - `매장(Store)` –1:N– `직원(Employee)`  
      → `Employee` 테이블에 `storeId FK` 추가.
- **N:M 관계 구현**
  - 항상 **중간 테이블(교차 테이블, 관계 테이블)** 을 만든다.
  - 예:
    - `Student` –N:M– `Course`
    - `Enrollment(studentId FK, courseId FK, ...)`
### 4. “저장의 관점에서 생각해야 함”
- ERD 설계 시 중요한 포인트:
  - **업무 상 관계**가 있다고 해서 모두 DB에 **저장해야 하는 관계**는 아님.
  - 예:
    - `고객`과 `FAQ`, `공지사항`:
      - 고객이 단지 화면에서 FAQ/공지사항을 **조회**만 한다면  
        → 항상 엔티티 간 직접 관계(선)를 그릴 필요는 없음.
      - “조회 이력”을 저장하고 싶다면, 별도의 로그 테이블을 설계.
- 결론:
  - ERD는 “어떤 정보를 **저장**하고, 그 정보들 사이를 어떻게 **연결해서 관리**할 것인가?”라는 **저장의 관점**에서 그려야 한다.
  - **저장할 필요가 있는 데이터/관계만 엔티티·관계로 모델링**하는 것이 핵심.
### 5. 오늘 ERD에서 얻은 실전 설계 감각
- **엔티티, 관계, 속성을 시각적으로 분리해서 생각하는 연습.**
- **관계 차수(1:1, 1:N, N:M)를 명확히 표현**하고 이해하기.
- **N:M 관계는 반드시 중간 테이블로 풀어내는 패턴** 익히기.
- “고객 ↔ FAQ”처럼 눈에 보이는 흐름이라도,  
  **실제로 무엇을 저장할지 기준으로 엔티티/관계를 결정**하는 사고 습관 들이기.
