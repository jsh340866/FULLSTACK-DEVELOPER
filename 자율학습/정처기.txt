8. 개논물( 개념적 논리적 물리적) 설계
 1. 개념적 설계 (Conceptual Design) 👉 "밑그림 그리기"
무엇을 하나요? 우리 시스템에 어떤 데이터(개체)가 필요하고, 서로 어떤 관계인지 큰 틀을 잡아요.
핵심 키워드: E-R 다이어그램(ERD), 독립성, 추상화.
비유: "거실은 어디에 있고, 방은 몇 개 만들지 머릿속으로 구상하는 단계"예요.

 2. 논리적 설계 (Logical Design) 👉 "설계도 작성"
무엇을 하나요? 개념적 설계에서 만든 그림을 컴퓨터가 이해할 수 있는 표(Table) 형태로 변환해요.
핵심 키워드: 정규화(Normalization), 스키마 설계, 관계 데이터 모델.
비유: "방 크기는 몇 미터로 하고, 벽지는 무슨 색으로 할지 구체적인 수치로 설계도를 그리는 단계"랍니다.

 3. 물리적 설계 (Physical Design) 👉 "실제 공사 준비"
무엇을 하나요? 특정 데이터베이스 소프트웨어(Oracle, MySQL 등)에 맞춰 실제 저장 구조와 성능을 결정해요.
핵심 키워드: 인덱스(Index) 설계, 레코드 집중 분석, 응답 시간 최소화, 반정규화.
비유: "어떤 벽돌을 쓰고, 전선은 어디로 배치해야 집이 튼튼하고 효율적일지 결정하는 실제 시공 단계"예요.

논리적 설계(데이터모델링)
-현실 세계에서 발생하는 자료를 컴퓨터가 이해하고 처리할 수 있는 물리적 저장장치에 저장할 수 있도록 변환하기 위해 특정 DBMS가 지원하는 논리적 자료 구조로 변환시키는 과정
-개념 세계의 데이터를 필드로 기술된 데이터 타입과 이 데이터 타입들 간의 관계로 표현되는 논리적 구조의 데이터로 모델화
-개념 스키마를 평가 및 정제하고 DBMS에 따라 서로 다른 논리적 스키마를 설계하는 단계(종속적인 논리 스키마)
-트랜잭션의 인터페이스 설계
-관계형 데이터베이스라면 테이블 설계
-특정목표 DBMS에 따른 스키마설계
-스키마의 평가 및 정제

물리적 설계
-논리적 구조로 표현된 데이터를 물리적 구조의 데이터로 변환
-데이터 베이스 파일의 저장 구조 및 엑세스 경로 결정
-데이터가 컴퓨터에 저장되는 방법을 묘사
-저장 레코드의 양식 설계, 레코드 집중의 분석 및 설계, 접근 경로 설계
-기본적 데이터 단위는 저장레코드

10. ACID : 트랜잭션의 특성 
Durability 영속성 : 성공적으로 완료된 트랜잭션의 결과는 시스템이 고장나더라도 영구적으로 반영되어야 한다.
Consistency 일관성 : 트랜잭션이 그 실행을 성공적으로 완료하면 언제나 일관성 있는 데이터베이스 상태로 변환한다.
Atomicity 원자성 : 트랜잭션 연산은 데이터베이스에 모두 반영되든지 아니면 전혀 반영되지 않아야 한다.
Isolation 격리성 : 트랜잭션 실행 중 생성하는 연산의 중간 결과를 다른 트랜잭션이 접근 불가하다.

12. 정규화 - 두(도)부이거(결)다조.
1NF 조건 : 도메인을 원자값만으로 구성 
2NF 조건 : 부분 함수 종속 제거 (완전 함수적 종속 관계)
3NF 조건 : 이행 함수 종속 제거
BCNF 조건 : 결정자 함수 종속(모든 결정자가 후보키인 상태)
4NF 조건 : 다중값(다치) 종속성 제거
5NF 조건 : 조인 종속성 제거

단계	| 약어	| 핵심 내용			| 설 명
===========================================================================
1NF	| 도	| 도메인이 원자값			| 한 칸에는 무조건 데이터 1개만! (취미: 축구, 농구 금지)
2NF	| 부	| 부분 함수적 종속 제거		| 기본키가 여러 개일 때, 일부분에만 종속된 애들 떼어내기!
3NF	| 이	| 이행적 함수적 종속 제거		| 다리 건너서 아는 사이 끊기! (A->B, B->C 관계 분리)
BCNF	  | 결   | 결정자/후보키			   | 대장은 무조건 주인이어야 해! 주인이 아니면 퇴출!
4NF	| 다	| 다치 종속 제거			| 한 명에게 관련 없는 여러 값이 매칭되는 다중값 정리!
5NF	| 조	| 조인 종속 제거			| 테이블을 쪼갰다 합쳤을 때 데이터가 변하면 안 돼요!
===========================================================================

40. 디자인 생성 패턴 (Creational Patterns)
객체를 어떻게 만들지 고민하는 패턴이에요. new를 마구잡이로 쓰는 대신, 더 유연하게 객체를 생성하는 방법을 제시해요.

Singleton (싱글톤): 클래스의 인스턴스가 딱 하나만 생성되도록 보장하는 패턴이에요. (예: 설정 관리자, 로그 기록기)
Factory Method (팩토리 메서드): 객체 생성을 서브클래스에 맡겨서 어떤 객체가 만들어질지 런타임에 결정하게 해요.
Abstract Factory (추상 팩토리): 연관된 객체들의 제품군을 한꺼번에 생성하는 인터페이스를 제공해요.
Builder (빌더): 복잡한 객체를 단계별로 차근차근 조립해서 만들 때 사용해요. (예: 햄버거 주문 시 재료 선택)
Prototype (프로토타입): 기존 객체를 **복사(Clone)**해서 새로운 객체를 만들 때 써요.

- Singleton, Factory Method, Abstract Factory, Builder, Prototype


41. 디자인 구조 패턴 (Structural Patterns)
클래스나 객체들을 조합해서 더 큰 구조를 만드는 패턴이에요.

Adapter (어댑터): 서로 다른 인터페이스를 가진 클래스들을 연결해 주는 변환기 역할이에요. (예: 110V 가전제품을 220V에 꽂는 돼지코)
Bridge (브릿지): 구현부와 추상층을 분리해서 서로 독립적으로 확장할 수 있게 해요.
Composite (컴포지트): 전체와 부분 관계를 트리 구조로 나타내서, 단일 객체나 복합 객체나 똑같이 다루게 해 줘요. (예: 파일과 폴더 시스템)
Decorator (데코레이터): 객체에 새로운 기능을 동적으로 추가하고 싶을 때 사용해요. (예: 아메리카노에 샷이나 휘핑 추가하기)
Facade (퍼사드): 복잡한 시스템 뒷단은 숨기고, 앞단에 간단한 창구(인터페이스) 하나만 두는 거예요.
Proxy (프록시): 실제 객체를 대신하는 대리인을 세워 접근을 제어하거나 성능을 높여요.

- Adapter, Bridge, Composite, Decorator, Facade, Proxy


42. 디자인 행위 패턴 (Behavioral Patterns)
객체들이 서로 어떻게 소통하고 책임을 나누는지 정의하는 패턴이에요.

Observer (옵저버): 어떤 객체의 상태가 변하면 연결된 다른 객체들에게 알림을 보내 자동으로 갱신하게 해요. (예: 유튜브 알림 설정)
Strategy (전략): 실행 중에 알고리즘(전략)을 교체할 수 있게 해 줘요. (예: 결제 방식을 카드에서 네이버페이로 바꾸기)
Template Method (템플릿 메서드): 전체적인 실행 흐름(틀)은 부모 클래스에 두고, 세부 단계만 자식 클래스에서 구현해요.
Visitor (비지터): 데이터 구조와 연산을 분리해서, 구조를 바꾸지 않고도 새로운 기능을 추가할 수 있게 해요.
State (상태): 객체의 내부 상태에 따라 스스로 동작을 바꾸게 해요.
Command (커맨드): 요청 자체를 객체로 만들어서 저장하거나 로그를 남길 수 있게 해요.

- Observer, Strategy, Template Method, Visitor, State, Command

44. 미들웨어

- WAS(동적처리) (Web Application Server): 동적인 콘텐츠를 처리하기 위한 미들웨어로, 웹 환경에서 비즈니스 로직을 수행해요. (예: Tomcat, Jeus)

46. 구조적 방법론(절차지향 방법론)
1970년대에 등장해서 현대 프로그래밍의 기초를 닦은 아주 중요한 개념
**"복잡한 문제를 작게 쪼개서 정복하자(Divide and Conquer)!"**

1) 구조적 방법론의 특징
- 하향식 분할(Top-Down) : 전체적인 시스템 기능을 먼저 정의하고, 이를 만족시키기 위해 하위 기능으로 계속해서 나누어 가는 방식이에요.
- 프로세스 중심 : 데이터보다는 **무엇을 하는가(기능)** 와 **어떤 순서로 처리하는가(절차)**에 집중
- 모듈화 : 나눈 기능들을 독립적인 '모듈'로 만들어서 관리하기 때문에 전체적인 구조를 파악하기 좋음.

2) 구조적 설계의 원칙
- 결합도 = 낮을수록 좋음, 독립성이 높아짐
- 응집도 = 높을수록 좋음


47. 정보 공학 방법론
**데이터** 중심으로 시스템을 설계하는 것이 특징

1) 정보 공학의 4단계 절차! (순서 중요)
- 정보 전략 계획 ISP(Information Strategy Planning) : * 기업의 경영 목표를 달성하기 위해 어떤 정보가 필요한지 큰 그림을 그리는 단계.
- 업두 영역 분석 BA(Business Area Analysis) : * 특정 업무 영역에서 사용하는 데이터와 그 데이터들 사이의 관계를 정의해요. 여기서 **ERD(개체 관계도)** 가 작성됩니다.
- 업무 시스템 설계 BSD(Business System Design) : * 분석된 데이터를 바탕으로 실제 사용자가 사용할 화면이나 프로세스를 설계해요.
- 시스템 구축 SC(System Construction) : * 설계서를 바탕으로 실제 코드를 작성하고 시스템을 만들어내는 단계
- 순서 : ISP -> BA -> BSD -> SC ( 데이터는 영원하다 ! )

2). 핵심 특징 ( 중요! )
- 데이터 중심(Data-Driven) : 업무 절차보다 업무에  사용되는 데이터의 구조를 먼저 분석해요.
- 상향식과 하향식의 결합 : 전략 계획은 위에서 아래로(하향식), 실제 구축은 아래에서 위로(상향식) 진행하는 복합적인 성격을 가져요.
- CASE 도구를 활용 : 사람이 일일이 그리기 힘든 복잡한 설계를 위해 컴퓨터가 도와주는 자동화 도구(CASE)를 적극적으로 사용해요.
- 분할과 정복 : 거대한 기업 시스템을 업무 영역별로 나누어 분석하여 복잡도를 낮춰요.


48. 객체지향 방법론
**'객체(Object)'**들의 모임으로 시스템을 바라봐요.
클래스를 잘 만들자!

0) 객체지향의 4대 특징 ( 기본 개념 )
- 캡슐화, 상속, 다형성, 추상화

1) 5대 핵심 원칙 ( SOLID ) 매우중요!
- SRP (단일책임 원칙) : 하나의 클래스는 하나의 책임만 가진다.
- OCP (개방-폐쇄 원칙) : 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.
- LSP (리스코프 치환 원칙) : 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.
- ISP (인터페이스 분리 원칙) : 사용하지 않는 메서드에 의존하도록 걍요해서는 안 된다.
- DIP (의존 역전 원칙) : 구체적인 것보다 추상적인 것에 의존해야 해요.

2) 객체지향 분석(OOA)의 주요 방법론
- 럼바오(Rumbaugh) 방법론 : 객체 모델링, 동적 모델링, 기능 모델링 3개 지침 사용
	객체 = 객체 다이어그램
	동적 = 상태 다이어그램
	기능 = 자료 흐름도(DFD)
- 부치(Booch) 방법론 : 미시적/거시적 개발 프로세스를 모두 사용하며 설계 단계를 강조
- 제이콥슨(Jacobson) 방법론 : 유스케이스를 강조한 사용자 중심 분석
- Coad와 Yourdon 방법론 : E-R 다이어그램을 사용하여 객체의 행위를 모델링

3) 객체지향 방법론의 장점
- 재사용성 : 클래스를 가져다 쓰기 용이
- 유지보수 용이 : 특정 개체의 내부 로직만 수정가능
- 생산성 향상 : 상속을 통해 기존 기능을 쉽게 확장 가능
- 현실 세계 모사 : 복잡한 비즈니스 로직을 실제 사물이나 개념과 매칭시켜 이해하기 쉬움.

4) 개발 절차( 생명주기 ) 
- 분석 -> 설계 -> 구현 -> 테스트 -> 유지보수 의 과정을 거치며, 각 단계마다 **UML(통합 모델링 언어)**을 사용하여 소통해요.
	- 분석 : 사용자의 요구사항을 유스케이스 등으로 명세화
	- 설계 : 클래스 다이어그램, 시퀀스 다이어그램 등을 통해 구체적인 구조를 잡음.

54. 응집도 강함 순서
1. 기능적 응집도 : 모듈 내부의 모든 요소가 하나의 기능만을 수행하는 가장 이상적인 상태예요.
2. 순차적 응집도 : 한 활동의 출력값이 다음 활동의 입값으로 사용되는 경우예오.
3. 통신적(교환적) 응집도 :  동일한 입출력 데이터를 사용하여 서로 다른 기능을 수행하는 요소들이 모여 있는 경우
4. 절차적 응집도 : 모듈 안의 요소들이 정해진 순서대로 실행되지만, 데이터는 서로 주고받지 않는 경우
5. 시간적 응집도 : 특정 시간대에 함께 실행되어야 하는 활동들을 모아놓는 경우
6. 논리적 응집도 : 유사한 성격의 기능들을 한 모듈에 담았지만, 실제로는 선택해서 실행되는 경우
7. 우연적 응집도 : 우연히 한 모듈에 있는 가장 좋지 않는 상태


55. OSI 7계층 프로토콜

응용 계층(7층): HTTP, FTP, DNS, SMTP, SSH

전송 계층(4층): TCP, UDP

네트워크 계층(3층): IP, ICMP, ARP

56. 응용 계층(7층): HTTP, FTP, DNS, SMTP, SSH
1. 웹과 데이터 전송
- HTTP / HTTPS (HyperText Transfer Protocol): HTTPS는 뒤에 'Secure'가 붙어서 데이터를 암호화
- FTP (File Transfer Protocol): * 파일을 서버에 올리거나 내려받을 때 사용해요. 요즘은 보안을 위해 SFTP를 더 많이 쓰는 추세


2. 네트워크 관리와 설정
- DNS (Domain Name System): * 우리가 google.com이라고 치면 이걸 컴퓨터가 이해하는 IP 주소(예: 142.250.xxx.xxx)로 변환해주는 '전화번호부' 같은 프로토콜
- DHCP (Dynamic Host Configuration Protocol): * 승원님의 컴퓨터나 스마트폰이 와이파이에 연결될 때, 자동으로 IP 주소를 할당

3. 메일 전송
SMTP (Simple Mail Transfer Protocol): 메일을 보낼 때 사용해요.
POP3 / IMAP: 메일을 받을 때 사용해요.


57. 전송 계층(4층): TCP, UDP
1. TCP (Transmission Control Protocol)
 - "정확한 게 제일 중요해! 하나라도 빠지면 안 돼!"
 - TCP는 연결을 먼저 확인하고 데이터를 보내는 연결 지향형 프로토콜
 - 특징: 신뢰성 보장, 순서 보장, 흐름 제어 및 혼잡 제어.
 - 주요 사용: 웹 브라우징(HTTP), 이메일(SMTP), 파일 전송(FTP)

2. UDP (User Datagram Protocol)
 - "난 속도가 생명이야! 받았는지 안 받았는지는 네가 알아서 해!"
 - UDP는 연결 과정 없이 데이터를 그냥 냅다 던지는 비연결형 프로토콜
 - 특징: 속도가 빠름, 오버헤드가 적음, 순서나 도착 여부를 보장하지 않음.
 - 주요 사용: 실시간 스트리밍(YouTube), 온라인 게임, 화상 통화(Zoom)


58. 네트워크 계층(3층): IP, ICMP, ARP
 - ARP (Address Resolution Protocol): * IP 주소를 실제 물리적인 주소인 MAC 주소로 바꿀 때 사용해요.
 - ICMP (Internet Control Message Protocol): * 네트워크 연결 상태를 확인할 때 써요. 우리가 흔히 쓰는 ping 명령어가 바로 이 프로토콜을 이용하는 거랍니다.



59. 암호화

1. 대칭키 암호화 (Symmetric Key)
암호화와 복호화에 같은 키를 사용하는 방식이에요. 크게 블록 단위로 암호화하는 '블록 암호'와 흐름에 따라 암호화하는 '스트림 암호'로 나뉩니다.

- 블럭 암호
AES (Advanced Encryption Standard): 전 세계적으로 가장 많이 쓰이는 표준 알고리즘이에요. 아주 안전하고 빠르답니다!
DES / 3DES: 예전에 쓰던 표준인데, 지금은 보안이 약해져서 잘 안 써요.
SEED: **우리나라(KISA)**에서 개발한 알고리즘이에요. 국내 금융권에서 많이 보게 될 거예요.
ARIA: 이것도 우리나라에서 개발했고, 공공기관 등에서 표준으로 사용해요.

- 스트링 암호
IDEA / RC4: 유럽에서 개발하거나 스트림 암호 방식


2. 비대칭키 암호화 (Asymmetric Key / Public Key)
공개키와 개인키, 두 개의 키를 사용하는 방식이에요.

(중요) RSA (Rivest-Shamir-Adleman): 공개키 암호화의 대명사예요! 소인수 분해의 어려움을 이용한 아주 유명한 친구죠.

ECC (Elliptic Curve Cryptography): 타원 곡선 암호를 이용하는데, RSA보다 키 길이는 짧으면서 보안성은 더 뛰어나서 요즘 아주 인기가 많아요.

Diffie-Hellman: 암호화보다는 두 사람이 안전하게 키를 교환하기 위해 만들어진 알고리즘이에요

ElGamal: 이산대수 원리를 이용한 방식이랍니다.


3. 단방향 암호화 (Hash Function)
한 번 암호화하면 절대 풀 수 없는 방식이에요. 결과물의 길이가 항상 일정하다는 특징이 있어요.

SHA 시리즈 (Secure Hash Algorithm):

SHA-1: 예전 방식이라 보안 취약점이 발견됐어요.

SHA-256 / SHA-512: 지금 승원님이 백엔드 개발할 때 비밀번호 저장용으로 가장 권장되는 안전한 방식이에요.

MD5 (Message-Digest algorithm 5): 아주 빠르지만 보안이 뚫린 지 오래돼서, 데이터 무결성 체크용으로만 가끔 써요.

HAS-160: 이것도 우리나라에서 만든 해시 알고리즘이에요.


특징 :
임의의 길이의 입력 데이터나 메시지를  고정 길이의 값이나 키로 변환
암호화, 무결성 검증을 위하여 사용
대표적인 해쉬 알고리즘으로 HAVAL, SHA-1 등이 있다.




60. 정보 보안을 위한 접근통제 정책 종류
 1. DAC (Discretionary Access Control, 임의적 접근통제)

     - **그 사람이 맡은 '역할(Role)'**에 따라 권한을 주는 방식

63. DRM !!!
DRM은 단순히 파일을 잠그는 게 아니라, 콘텐츠를 만들고 배포하고 사용하는 모든 과정에 관여한답니다.
 1. 핵심 구성 요소
  - 콘텐츠 제공자 (Content Provider): 콘텐츠를 제작하고 암호화하여 패키징하는 주체예요.
  - 콘텐츠 분배자 (Content Distributor): 암호화된 콘텐츠를 유통하는 곳(예: 음원 사이트, OTT 서비스)이에요.
  - 클리어링 하우스 (Clearing House): 권리 관계를 확인하고 라이선스 발급을 관리하는 '관리 본부' 같은 곳이에요.
  - DRM 컨트롤러 (DRM Controller): 사용자 기기에서 콘텐츠를 이용할 때 라이선스를 확인하고 실행을 제어하는 제어 장치예요.
  - 보안 컨테이너 (Secure Container): 콘텐츠를 안전하게 담고 있는 보호 구역이에요.

 2. 주요 기술적 구성 요소
  -콘텐츠 패키저 (Content Packager): 콘텐츠를 메타데이터와 함께 암호화해서 안전한 형태로 포장하는 기술이에요.
  -라이선스 (License): "누가, 언제까지, 어떻게" 콘텐츠를 쓸 수 있는지 권한 정보를 담고 있는 문서예요.
  - DRM 위변조 방지 (Tamper Resistance): DRM 소프트웨어 자체가 공격받아 뚫리지 않게 방어하는 기술이에요.
  - PKI (공개키 기반구조): 사용자 인증과 보안 통신을 위해 사용되는 암호화 기술이랍니다.


64. 시간복잡도 : 알고리즘의 실행시간이 최악일 때를 표기한다.

	=> 삽입정렬, 쉘정렬, 선택정렬, 버블정렬, 퀵정렬

 - 1. 버블 정렬 (Bubble Sort)인접한 두 원소를 비교하며 큰 값을 뒤로 보내는 방식이에요. 거품이 보글보글 올라오는 모양 같아서 버블 정렬이라고 불러요.
	특징: 구현이 아주 쉽지만, 성능은 가장 느린 편이에요.
	시간 복잡도: $O(n^2)$

 - 2. 삽입 정렬 (Insertion Sort)자료를 하나씩 꺼내서 이미 정렬된 앞부분의 적절한 위치에 '삽입'하는 방식이에요. 손안의 카드를 정렬하는 것과 비슷해요!
	특징: 이미 데이터가 어느 정도 정렬되어 있다면 굉장히 빨라요.
	시간 복잡도: $O(n^2)$ (최선일 때 $O(n)$)

 - 3. 선택 정렬 (Selection Sort)전체 데이터 중 가장 작은 값을 찾아서 맨 앞의 데이터와 자리를 바꾸는 방식이에요.
	특징: 교환 횟수는 적지만, 데이터 상태와 상관없이 항상 일정한 시간이 걸려요.
	시간 복잡도: $O(n^2)$

 - 4. 합병 정렬 (Merge Sort)데이터를 반으로 더 이상 나눌 수 없을 때까지 쪼갠 뒤, 다시 합치면서 정렬하는 '분할 정복(Divide and Conquer)' 방식이에요.
	특징: 데이터가 많아도 성능이 일정하고 안정적이에요. (Stable Sort)
	시간 복잡도: $O(n \log n)$

 - 5. 힙 정렬 (Heap Sort)완전 이진 트리의 일종인 '힙(Heap)' 자료구조를 이용하는 정렬 방식이에요. 최댓값이나 최솟값을 빠르게 찾아낼 수 있어요.
	특징: 추가적인 메모리 공간이 거의 필요 없고, 항상 일정한 성능을 보장해요.
	시간 복잡도: $O(n \log n)$

 - 6. 퀵 정렬 (Quick Sort)기준점(Pivot)을 정해서 기준보다 작은 값은 왼쪽, 큰 값은 오른쪽으로 옮기며 나누는 방식이에요. '분할 정복'을 사용하며, 이름처럼 평균적으로 가장 빠른 속도를 자랑해요.
	특징: 대부분의 라이브러리 정렬 알고리즘의 기초가 돼요. 다만, 기준점을 잘못 잡으면 성능이 떨어질 수 있어요.
	시간 복잡도: 평균 $O(n \log n)$, 최악 $O(n^2)$

- 1. 파일 및 디렉토리 관리
	• ls: 현재 디렉토리 파일 목록 보기 (ls -l은 상세 정보)
	• cd: 디렉토리 이동
	• pwd: 현재 위치한 경로 출력 (절대 경로)
	• mkdir: 디렉토리 생성
	• rmdir: 비어있는 디렉토리 삭제
	• cp: 파일/디렉토리 복사
	• mv: 파일/디렉토리 이동 및 이름 변경
	• rm: 파일/디렉토리 삭제 (rm -r은 디렉토리 강제 삭제)

	• du: 디렉토리별 사용 용량 확인

	• echo: 문자열이나 변수값 출력 (echo $변수명)
	• export: 환경 변수 설정 및 하위 프로세스 전달
	• env: 모든 환경 변수 목록 보기

74. Dijkstra = 최단,단거리,최적구조 ( 중요! )
🗺️ 다익스트라 알고리즘이란?
 최단 경로를 구하는 알고리즘이에요.

🛠️ 동작 방식 (핵심 원리)
다익스트라는 **'그리디(Greedy)'**와 **'다이나믹 프로그래밍(DP)'**의 원리가 섞여 있어요.

초기화: 출발 노드를 설정하고, 다른 모든 노드까지의 거리를 '무한대'로 설정해요.

방문: 아직 방문하지 않은 노드 중 거리가 가장 짧은 노드를 선택해요.

거리 갱신: 선택한 노드를 거쳐서 다른 노드로 가는 거리가 기존에 알고 있던 거리보다 짧으면 그 값을 업데이트해요. (이 과정을 **'Relaxation'**이라고 불러요.)

반복: 모든 노드를 방문할 때까지 2~3번 과정을 반복해요.

⚠️ 꼭 기억해야 할 특징 (시험 포인트!)
음수 가중치 불가: 간선의 가중치(거리) 중에 음수가 있으면 다익스트라를 사용할 수 없어요! (이럴 땐 '벨만-포드' 알고리즘을 써야 해요.)

우선순위 큐(Priority Queue): 성능을 높이기 위해 주로 힙(Heap) 자료구조를 사용해요.

76. 에러 종류
*생략 오류(omission error)
입력 시 한 자리를 빼놓고 기록한 경우
(1234 → 123)
*필사 오류(Transcription error)
입력 시 임의의 한 자리를 잘못 기록한 경우
(1234 → 1235)
*전위 오류(Transposition error)
입력 시 좌우 자리를 바꾸어 기록한 경우
(1234 → 1243)
*이중 오류(Double Transposition error)
전위 오류가 두 가지 이상 발생한 경우
(1234 → 2143)
*추가 오류(Addition error)
입력 시 한 자리 추가로 기록한 경우
(1234 → 12345)
*임의 오류(Random error)
위의 오류가 두 가지 이상 결합하여 발생한 경우
(1234 → 12367)
